# Event Management Service

REST API на ASP.NET Core для управления событиями и бронированиями.

## Возможности

- Создание, получение, обновление, удаление и фильтрация событий.
- Создание брони для события.
- Получение текущего состояния брони.
- Фоновая обработка бронирований в статусе `Pending`.
- In-memory хранение данных без внешней базы.
- Swagger/OpenAPI для просмотра и ручной проверки API.

## Модель события

Событие имеет идентификатор типа `Guid`.

Основные поля:

- `id` - `Guid`
- `title` - название события
- `description` - описание
- `startAt` - дата и время начала
- `endAt` - дата и время окончания

## Модель брони

`Booking` описывает бронь на событие.

Поля:

- `id` - `Guid`, уникальный идентификатор брони
- `eventId` - `Guid`, идентификатор события
- `status` - текущий статус брони
- `createdAt` - дата и время создания
- `processedAt` - дата и время обработки, может быть `null`

Статусы `BookingStatus`:

- `Pending` - бронь создана и ожидает обработки
- `Confirmed` - бронь подтверждена
- `Rejected` - бронь отклонена

При создании бронь получает:

- уникальный `id`
- `status = Pending`
- текущую дату и время в `createdAt`
- пустой `processedAt`

## API событий

### Получить список событий

```http
GET /events
```

Query-параметры:

- `page` - номер страницы, по умолчанию `1`
- `pageSize` - размер страницы, по умолчанию `10`
- `title` - фильтр по названию
- `from` - фильтр по дате начала
- `to` - фильтр по дате окончания

### Получить событие по Id

```http
GET /events/{id}
```

Возвращает `200 OK`, если событие найдено.

Возвращает `404 Not Found`, если события нет.

### Создать событие

```http
POST /events
```

Пример тела запроса:

```json
{
  "title": "Music Event",
  "description": "Live concert",
  "startAt": "2026-07-01T10:00:00",
  "endAt": "2026-07-01T12:00:00"
}
```

Возвращает `201 Created`.

### Обновить событие

```http
PUT /events/{id}
```

Возвращает `200 OK`, если событие обновлено.

Возвращает `404 Not Found`, если события нет.

### Удалить событие

```http
DELETE /events/{id}
```

Возвращает `204 No Content`, если событие удалено.

Возвращает `404 Not Found`, если события нет.

## API бронирований

### Создать бронь для события

```http
POST /events/{id}/book
```

Где `{id}` - `Guid` события.

Если событие существует, создаётся бронь в статусе `Pending`.

Ответ:

- `202 Accepted`
- тело ответа содержит данные созданной брони
- заголовок `Location` содержит ссылку на ресурс брони: `/bookings/{bookingId}`

Пример ответа:

```json
{
  "id": "4c8f18d4-923a-495f-b9f0-2df0b51bd75d",
  "eventId": "7cdca0dc-c377-4d9d-9e50-ecf50726e358",
  "status": "Pending",
  "createdAt": "2026-06-24T08:00:00Z",
  "processedAt": null
}
```

Если событие не найдено, возвращается `404 Not Found`.

### Получить бронь по Id

```http
GET /bookings/{id}
```

Возвращает текущее состояние брони.

Ответы:

- `200 OK` - бронь найдена
- `404 Not Found` - бронь не найдена

## Фоновая обработка бронирований

В приложении зарегистрирован `BookingProcessingService` на базе `BackgroundService`.

Сервис работает в фоне после старта приложения:

1. Периодически получает брони со статусом `Pending`.
2. Для каждой брони ждёт 2 секунды, имитируя обращение к внешней системе.
3. Переводит бронь в статус `Confirmed`.
4. Заполняет `processedAt`.
5. Сохраняет обновлённую бронь в in-memory хранилище.

При остановке приложения `OperationCanceledException` обрабатывается явно, чтобы штатная остановка не попадала в логи как ошибка.

## Пример сценария

1. Создать событие:

```http
POST /events
```

2. Из ответа взять `id` события.

3. Создать бронь:

```http
POST /events/{eventId}/book
```

4. Из ответа взять `id` брони или значение заголовка `Location`.

5. Сразу проверить бронь:

```http
GET /bookings/{bookingId}
```

Статус будет `Pending`.

6. Подождать несколько секунд и повторить запрос:

```http
GET /bookings/{bookingId}
```

Статус станет `Confirmed`, а `processedAt` будет заполнен.
